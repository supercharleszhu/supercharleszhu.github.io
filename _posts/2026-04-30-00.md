## Why "how is a dep decided" is the right lens

The Inductor pipeline has a lot of moving parts — Dynamo, AOTAutograd, lowerings, the scheduler, fusion, memory planning, wrapper codegen, Triton kernel emission. Most write-ups walk the stages in order and mention dependency tracking as one item among many. I'm going to do the opposite: thread the **buffer / arg dependency story** through every stage, and use it as the spine that ties FX-graph nodes to the `del bufN` lines you see in the generated Python wrapper.

The reason is that almost every nontrivial Inductor bug — fused kernels reading freed buffers, in-place reuse clobbering live data, mismatched signatures, peak memory regressions — comes from a layer's view of "what does this kernel read/write" disagreeing with the next layer's view. Pin down how that read/write set is computed at each stage and the rest of the pipeline reads off it cleanly.

I'll close with a real example: a CuteDSL flash-attention kernel hits exactly this dep-disagreement problem on a recent PyTorch nightly, and the wrapper helpfully emits `del buf180` immediately before a kernel that reads `buf180`. The bug is a one-paragraph case study; the rest of the post is the framework you need to even describe what went wrong.

## Stage 0 — From `torch.compile` to Inductor's front door

When you call `torch.compile(fn)(...)`, control flows: Dynamo traces Python bytecode into an FX graph, AOTAutograd runs functionalization and decompositions and splits forward / backward, then hands each half to a backend. With the default backend, that backend is Inductor.

Inductor's entry point is [`fx_codegen_and_compile`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/compile_fx.py#L1785), which receives a `torch.fx.GraphModule` and example inputs:

```python
def fx_codegen_and_compile(
    gm: GraphModule,
    example_inputs: Sequence[InputType],
    inputs_to_check: Sequence[int],
    compile_region_name: str | None = None,
    **graph_kwargs: Unpack[_CompileFxKwargs],
) -> OutputCode:
```

The FX graph at this boundary already has all autograd machinery resolved: nodes are `placeholder`, `call_function` (typically with `aten.*` or `prims.*` targets), and `output`. There are no `torch.nn.Module` calls, no autograd tape, no Python control flow — just a flat dataflow graph of ATen operations on fake tensors.

What Inductor does next is, broadly, two passes: an FX-walk that builds Inductor's own IR (the [`GraphLowering`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/graph.py#L364) pass), then a backend pass that schedules, fuses, and emits Python + Triton.

## Stage 1 — `GraphLowering`: FX → Inductor IR

`GraphLowering` is a [`torch.fx.Interpreter`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/graph.py#L364), so it walks the FX graph node-by-node, dispatching on node opcode. Three cases matter:

### `placeholder` — graph inputs become `InputBuffer`s

[`GraphLowering.placeholder`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/graph.py#L1217) registers each FX placeholder as an `InputBuffer`, names it with the FX node name (so `arg0_1`, `arg1_1`, …, in the order the placeholder appeared), and stores it in `self.graph_inputs`. These are the `argN_1` you see in the wrapper output.

The names are *load-bearing* — every downstream layer references buffers by these strings. Two consequences that bite later: graph inputs cannot have their storage reused as scratch (the wrapper has a special branch for this), and they have whole-graph lifetime by default.

### `call_function` — ATen ops become Inductor IR via lowerings

[`GraphLowering.call_function`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/graph.py#L1327) looks up the FX target in the global [`lowerings`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/lowering.py#L116) registry and dispatches:

```python
lowerings: dict[Callable[..., Any] | str, Callable[..., Any]] = {}
```

Lowerings are registered with the [`@register_lowering(...)`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/lowering.py#L529) decorator. There are hundreds — one per ATen op, give or take — and most are short. A representative pointwise op is built from [`make_pointwise`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/lowering.py#L669), which produces a `Pointwise` IR node (an opaque computation over an output index) that later becomes a `ComputedBuffer` once it's materialized.

Three key IR types come out of this stage, all defined in [`torch/_inductor/ir.py`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/ir.py):

* **`ComputedBuffer`** — a "fused-by-construction" loop body. Wraps a `Pointwise` or `Reduction` and a `Layout`. The body is a `LoopBody` you can symbolically interpret; this property is what makes its dependencies analyzable.
* **`TemplateBuffer`** — a kernel template. Includes [`TritonTemplateBuffer`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/ir.py#L5292) (built-in Triton templates like matmul), [`CuteDSLTemplateBuffer`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/ir.py#L5812), `HelionTemplateBuffer`, etc. The body is a Jinja template — *not* something the scheduler can interpret op-by-op.
* **`InputBuffer`** — graph inputs from `placeholder` (already discussed).

Plus auxiliary types: `ReinterpretView` (cheap view that shares storage with a source buffer), `MultiOutput` (one logical buffer feeding several named outputs), `MutationOutput` (in-place mutation marker), and so on.

When the lowering produces a buffer, Inductor calls `V.graph.register_buffer(buf)`. This is where the `bufNNN` names come from: the registry hands out monotonically increasing IDs. So the `buf180` in the bug case study is the 180-th internal buffer registered, in some FX traversal order.

### `output` — graph outputs are pinned

Finally [`GraphLowering.output`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/graph.py#L1559) records which buffers the compiled function will return. These names get added to `V.graph.get_output_names()` and become the seed for the scheduler's last-use analysis: outputs are live to the end, so they never get freed inside `call(...)`.

At this point the IR is a directed acyclic graph of Buffers. Each buffer has a name, a layout, and either a loop body (for `ComputedBuffer`) or a template binding (for `TemplateBuffer`).

## Stage 2 — The Scheduler: where dependencies become `read_writes`

The next pass wraps each operation buffer in a `BaseSchedulerNode` to form a scheduler graph. From here on, the unit of work is a *scheduler node*; the unit of memory is a *named buffer*; and the unit of dependency information is a `ReadWrites` object attached to each scheduler node.

A `BaseSchedulerNode` carries a [`read_writes: ReadWrites`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/scheduler.py#L558) attribute, which is a pair of ordered sets of `Dep`s. There are several `Dep` flavors — `MemoryDep` (typed, indexed access), `StarDep` (whole-buffer read/write), `WeakDep` (ordering-only, doesn't extend lifetime). The scheduler distinguishes them but for liveness purposes a name appearing in `read_writes.reads` or `.writes` keeps that buffer alive through this node.

How is `read_writes` populated? The fork is in [`SchedulerNode._compute_attrs`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/scheduler.py#L1554):

```python
if isinstance(self.node, ir.TemplateBuffer):
    self.set_read_writes(
        self.node.extract_read_writes(normalize=should_normalize)
    )
else:
    self.set_read_writes(
        dependencies.extract_read_writes(
            self._body, *self._sizes, normalize=should_normalize
        )
    )
```

This is the most important fork in the entire codegen pipeline.

### Path A: `ComputedBuffer` — symbolic interpretation of the loop body

For a `ComputedBuffer`, the loop body is a Python callable (after `simplify_and_reorder`) that takes loop indices and emits ops via a virtualized handler. `dependencies.extract_read_writes(...)` runs this callable under a tracking handler that records every `ops.load(name, idx)` as a read and every `ops.store(name, idx, ...)` as a write.

Result: a `ReadWrites` that **faithfully reflects every storage the kernel will touch**, including any captured tensor pulled in via `ops.load` — because the only way a captured tensor enters the loop body is via a load.

Aliases get handled with one explicit walk: [`used_or_aliased_buffer_names()`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/scheduler.py#L696) extends each read with `get_inputs_that_alias_output()` so a `reinterpret_tensor(buf180, …)` read keeps `buf180` alive. (`is_fake` `WeakDep`s are filtered out — they exist purely to constrain ordering.)

### Path B: `TemplateBuffer` — the inputs-list shortcut

For a `TemplateBuffer`, the body is a Jinja template. There's no Python loop body to interpret. Instead, the template buffer itself is asked: [`TemplateBuffer.extract_read_writes`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/ir.py#L5378) builds reads from [`_read_deps_from_inputs`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/ir.py#L5360):

```python
def _read_deps_from_inputs(self, normalize: bool) -> OrderedSet[dependencies.Dep]:
    """Build read dependencies from all inputs."""
    reads: OrderedSet[dependencies.Dep] = OrderedSet()
    for inp_raw in self.inputs:
        ...
        reads |= dependencies.extract_read_writes(
            dummy, inp.get_size(), (), normalize=normalize
        ).reads
    return reads
```

That `for inp_raw in self.inputs:` is the load-bearing line. **Whatever the template kernel actually reads beyond its declared `inputs` list is invisible to the scheduler.**

For a Triton template that's mostly fine: epilogue / prologue fusion is implemented as separate scheduler nodes that retain their own loop bodies and `read_writes`, then get folded into the final fused node. The fold preserves their reads. Captures of "outer" tensors aren't a thing for built-in Triton matmul-style templates.

For a CuteDSL template — particularly the flash-attention variant in [`torch/_inductor/kernel/flex/flex_flash_attention.py`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/kernel/flex/flex_flash_attention.py) — captures of outer tensors *are* a thing: the user's `score_mod` and `mask_mod` Python callables can reference any tensor they want, and the cutedsl kernel signature is built later by walking those subgraphs for free variables. Those captures never appear in `self.inputs`, so the SchedulerNode's read set undercounts the kernel's true reads.

This is exactly the failure mode we'll see at the end of the post. Hold onto it.

### `set_read_writes`, `used_buffer_names`, and friends

A few related plumbings keep `read_writes` accurate as the scheduler manipulates it:

* [`set_read_writes`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/scheduler.py#L674) is the canonical setter; it always updates `unmet_dependencies` to `read_writes.reads` so dependency-resolution stays in sync.
* [`used_buffer_names`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/scheduler.py#L690) flattens `reads` and `writes` to a name set.
* [`used_or_aliased_buffer_names`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/scheduler.py#L696) extends with one alias hop.
* `prune_deps()` removes deps to ops that have been DCE'd or removed — it edits `read_writes` directly via `set_read_writes(read_writes.remove_reads(to_remove))`.

Everything downstream that asks "what does this node touch?" — fusion, last-use analysis, peak memory estimation, kernel signature emission — reads from these. Get `read_writes` wrong at the source and the rest of the pipeline lies happily.

## Stage 3 — Fusion: collapsing scheduler nodes

After basic scheduler graph construction, Inductor runs a fusion pass that collapses multiple `SchedulerNode`s into `FusedSchedulerNode`s. The pass is iterative: it tries every adjacent pair (vertical fusion = producer + consumer) or every pair sharing a parent (horizontal fusion = sibling consumers of the same producer), checks whether the backend can fuse them, and repeats until no more fusions are profitable.

The backend gates this. For Triton-eligible nodes, [`SIMDScheduling.can_fuse`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/codegen/simd.py#L1308) checks shape compatibility, reduction kinds, and a long list of "would this be sound and profitable" predicates. The result is a fused node that exposes a `read_writes` covering the union of its children's reads/writes minus any *internally satisfied* deps (a buffer produced and consumed within the same fusion is no longer a public read/write).

[`FusedSchedulerNode.set_last_usage`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/scheduler.py#L2095) does a two-level pass: first the global `set_last_usage` on the fused node (so other nodes see the fused boundary), then a reverse pass over the inner `snodes` to compute per-snode last-use. The inner pass is what enables Triton kernels to drop a register the moment a value is no longer needed inside the kernel body.

The fork still applies after fusion: a fused node wrapping a `TemplateBuffer` still has its `read_writes` derived from the template's `_read_deps_from_inputs`, plus whatever epilogue snodes were folded in via their own loop bodies. The captures-not-in-inputs problem doesn't get fixed by fusion.

## Stage 4 — Memory planning and last-use analysis

This is where the pipeline turns "node N reads buffer B" into "emit `del B` at line K of the wrapper output."

### `compute_last_usage`

The function is short. From [`scheduler.py:6811`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/scheduler.py#L6811):

```python
def compute_last_usage(self) -> None:
    """
    Populate node.last_usage recursively (also for the nodes within a
    FusedSchedulerNode)
    """
    future_used_buffers = OrderedSet(V.graph.get_output_names())

    for node in reversed(self.nodes):
        node.set_last_usage(future_used_buffers, self.mutation_real_name)
        future_used_buffers.update(node.last_usage)
```

The recipe in plain English: walk in reverse execution order; seed the "live set" with whatever the function returns (those have to stay live to the end); for each node, compute `last_usage = used_or_aliased_buffer_names(self) - future_used_buffers`; then add those names to `future_used_buffers` so earlier nodes see them as live.

[`set_last_usage`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/scheduler.py#L679) implements the per-node piece:

```python
def set_last_usage(self, future_used_buffers, mutation_real_name) -> None:
    used_buffers = self.used_or_aliased_buffer_names()
    used_buffers = OrderedSet(mutation_real_name.get(k, k) for k in used_buffers)
    self.last_usage = used_buffers - future_used_buffers
```

The `mutation_real_name` step renames in-place mutated buffers to their canonical name so a kernel that mutates `buf3` in place isn't treated as keeping `buf2` (the pre-mutation alias) alive past its real death.

### Peak memory and reuse planning

Beyond last-use, [`torch/_inductor/memory.py`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/memory.py) builds memory-planning information. [`assign_memory_planning_info_for_scheduler_buffers`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/memory.py#L226) annotates every buffer with its size and, importantly, a list of "successors who use this buffer." [`estimate_peak_memory_allocfree`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/memory.py#L479) walks the schedule with this info, simulating allocations and frees, to produce the peak-memory estimate that drives reuse planning and topological re-ordering ([`topological_sort_lpmf`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/memory.py#L540)).

Reuse is the optimization where, when buffer A dies on the same node where buffer B is born, the wrapper rebinds B's name to A's storage instead of allocating fresh memory: `bufB = bufA; del bufA  # reuse`. This whole pass — what's reusable, what reuse saves, where to put `del` lines — runs on top of `last_usage`, which runs on top of `read_writes`. **Every layer in this stack inherits the dep set computed in Stage 2.**

## Stage 5 — Wrapper codegen: the visible artifact

Now we get to the Python file you can actually read in `/tmp/torchinductor_*/.../output_code.py`.

[`GraphLowering.codegen`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/graph.py#L2555) hands control to a `PythonWrapperCodegen` (or `CppWrapperCpu` for AOT-Inductor C++ output). The wrapper iterates the scheduler in execution order. For each node, it asks the relevant backend to codegen the kernel (Stage 6 below), then emits:

* **Allocation lines.** `bufN = empty_strided_cuda(size, stride, dtype)` if the buffer needs fresh storage, or `bufN = reinterpret_tensor(bufM, …)` for views, or `bufN = bufM; del bufM  # reuse` for reuse rebinds.
* **The kernel call.** `<kernel_name>.run(arg, arg, …, stream=streamN)`. The argument list comes directly from the kernel's emitted signature (Stage 6).
* **Free lines.** One `del` per name in the node's `last_usage`, mediated by [`codegen_free`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/codegen/wrapper.py#L3691):

```python
def codegen_free(self, buffer):
    name = buffer.get_name()

    if isinstance(buffer, (ir.InputBuffer, ir.TorchBindObject)):
        self.writeline(FreeLine(self, buffer))   # always del
        return

    if isinstance(buffer.get_output_spec(), ir.CommBufferLayout):
        self.writeline(FreeIfNotReusedLine(self, buffer, comm_buffer=True))
        return

    if not self.can_reuse(buffer):
        return
    self.freed.add(name)
    self.writeline(FreeIfNotReusedLine(self, buffer))
```

Two flavors:

* `FreeLine` for graph inputs and TorchBind objects — always emit `del argN_1`. The wrapper can't repurpose input storage as scratch, so there's no "maybe don't free" path.
* `FreeIfNotReusedLine` for internal `bufN` — the wrapper waits until rendering time to decide. If a later allocation has been planned to reuse this buffer's storage (reuse decision happens during the scheduler walk), no free line is emitted; otherwise, `del bufN`.

There's also a sweep at kernel boundaries: [`Scheduler.free_buffers`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/scheduler.py#L6822) drains a `buffer_names_to_free` queue, ensuring nothing lingers past its computed death.

A representative slice of generated wrapper output:

```python
def call(args):
    arg0_1, arg1_1, arg2_1 = args
    args.clear()
    with torch.cuda._DeviceGuard(0):
        torch.cuda.set_device(0)
        buf0 = empty_strided_cuda(...)
        triton_poi_fused__to_copy_view_0.run(arg0_1, buf0, ...)
        buf1 = empty_strided_cuda(...)
        triton_red_fused_sum_1.run(buf0, arg1_1, buf1, ...)
        del buf0
        del arg0_1
        return (buf1,)
```

Every `del` line comes from `last_usage` for the node above it; every `bufN = ...` comes from an allocation request emitted during scheduler walk; every `<kernel>.run(...)` argument list comes from the kernel's signature (next stage). All of which derive from the dep set computed in Stage 2.

## Stage 6 — Triton kernel codegen: where the dep set becomes `tl.load`

This is where the abstraction becomes physical. For Triton-eligible scheduler nodes, [`SIMDScheduling`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/codegen/simd.py#L1308) drives kernel emission via its [`codegen_node`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/codegen/simd.py#L1841) entry point, which assembles a [`TritonKernel`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/codegen/triton.py#L2789) (a subclass of [`SIMDKernel`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/codegen/simd.py#L388)) and calls `codegen_kernel`.

### The kernel signature is a function of `read_writes`

The Triton kernel's parameter list is built by walking the fused node's reads (input pointers), writes (output pointers), and any auxiliary scalars or sizes. The wrapper then matches that signature when emitting `<kernel>.run(...)`. The same `read_writes` from Stage 2 that drove last-use analysis also drives signature construction.

If `read_writes` is wrong — say, missing a captured tensor — *both* the wrapper's argument list at the call site *and* the kernel's parameter list would be missing that pointer, so they'd agree and the call would just compute on stale data. (That's a subtle silent-correctness issue in a different layer; the bug we'll close on is more visible because it causes a Python-level `UnboundLocalError`.)

### Tiling and indexing

[`SIMDKernel.codegen_indexing`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/codegen/simd.py#L1056) takes a sympy index expression and rewrites it into an "indexing variable" that the kernel can use; this is where strided loads turn into something like `tl.load(in_ptr0 + (x0 + 64 * x1))`. [`SIMDScheduling.select_tiling`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/codegen/simd.py#L3014) chooses how to split the iteration space across `XBLOCK` / `RBLOCK` (and sometimes `YBLOCK`) — i.e. it decides the kernel's pid axes.

Tiling decisions feed back into masks: a kernel processing `numel = 1234` with `XBLOCK = 1024` needs `xmask = xindex < xnumel` to avoid out-of-bounds loads on the tail block. The mask emission piggybacks on indexing.

### `codegen_body` and `codegen_kernel`

[`TritonKernel.codegen_body`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/codegen/triton.py#L5181) walks the loop body once more, this time emitting Triton ops (`tl.load`, `tl.store`, `tl.sum`, `tl.where`, etc.) into an `IndentedBuffer`. Reductions wrap their accumulators in `RBLOCK` loops; pointwise ops are flat. CSE (common subexpression elimination) deduplicates intermediate computations across snodes within the same kernel.

[`TritonKernel.codegen_kernel`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/codegen/triton.py#L5625) wraps it all into a `@triton.jit`-decorated function: `_get_args` produces typed argdefs, `jit_lines` emits the `@triton.jit` / `@triton_heuristics.pointwise(...)` decorators (with grid metadata, num_warps, num_stages, etc.), and the body is the buffer from `codegen_body`. The result is a string spliced into the generated Python file with a name like `triton_poi_fused__to_copy_add_clone_view_NN`.

A peek at what comes out:

```python
@triton_heuristics.pointwise(
    size_hints={'x': 32768},
    filename=__file__,
    triton_meta={...},
    inductor_meta={...},
)
@triton.jit
def triton_poi_fused__to_copy_add_clone_view_3(
    in_ptr0, in_ptr1, out_ptr0, xnumel, XBLOCK : tl.constexpr,
):
    xnumel = 26214400
    xoffset = tl.program_id(0) * XBLOCK
    xindex = xoffset + tl.arange(0, XBLOCK)[:]
    xmask = xindex < xnumel
    x0 = xindex
    tmp0 = tl.load(in_ptr0 + (x0), xmask)
    tmp1 = tl.load(in_ptr1 + (x0 % 4096), xmask)
    tmp2 = tmp0 + tmp1
    tl.store(out_ptr0 + (x0), tmp2, xmask)
```

Everything in here — `in_ptr0, in_ptr1, out_ptr0` — corresponds to a name in the wrapper's argument list, which corresponds to a buffer in `read_writes`, which corresponds to a name allocated by `register_buffer` back in Stage 1.

### Templates and CuteDSL

For `TemplateBuffer` nodes the path is different: [`SIMDScheduling.codegen_template`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/codegen/simd.py#L2190) renders a Jinja template and stitches it into the generated file, and CuteDSL templates go through a parallel path in `cuda_combined_scheduling.py` and `codegen/cutedsl/`. The kernel signature for these is composed from `self.inputs` plus a list of mutated/captured tensors fished out of subgraphs at render time. **This is where the Stage 2 invariant breaks for CuteDSL flash-attention** — the rendered kernel signature includes the captured mask tensor, but the scheduler-side `read_writes` does not.

## A real bug: when the dep set lies

A recent DV365 training run on PyTorch nightly `2.13.0.dev20260426+cu130` crashed in inductor-generated wrapper code with:

```
UnboundLocalError: cannot access local variable 'buf180'
where it is not associated with a value
```

The trace bottoms out in a generated `output_code.py` looking like:

```python
buf180 = empty_strided_cuda((3584, 2047), (2048, 1), torch.bool)
buf152 = reinterpret_tensor(buf180, ...)  # alias
# ... 27 more aliases of buf180 ...
triton_poi_fused_..._20.run(_tensor_constant3, buf180, arg104_1, ...)
triton_red_fused_..._21.run(buf180, arg104_1, ...)
triton_poi_fused_..._22.run(buf180, arg104_1, ...)
triton_red_fused_..._23.run(buf180, arg104_1, ..., buf250, buf274, ...)
del arg104_1
del arg5_1
del buf180                                      # <-- buffer freed here
# ... 130 lines of unrelated codegen ...
cutedsl_fused__to_copy_add_bitwise_and_clone_eq_flex_attention_..._53177e8e.run(
    ..., buf180, arg104_1, ..., arg5_1, ...    # <-- still consuming three deleted vars
)
```

A buffer is allocated, fed to four Triton fused kernels, marked dead, freed — and then 130 lines later a CuteDSL-backed `flash_attention` kernel reads it again. Same pattern for `arg104_1` and `arg5_1`. The model uses `flex_attention(..., kernel_options={'BACKEND': 'FLASH'})`, opting into the CuteDSL backend; `mask_mod` captures a `mask` tensor that becomes `buf180` in the lowered IR.

Map this onto the framework above:

1. **Stage 2.** The `CuteDSLTemplateBuffer` for the flash-attention call has `inputs = [q, k, v, kv_num_blocks, kv_indices, full_kv_num_blocks, full_kv_indices, output]`. The captured `mask` is *not* in `inputs` — it's wired into the kernel signature later when `cutedsl_kernel.py` walks the `mask_mod` subgraph for free variables. So the SchedulerNode's `read_writes.reads` doesn't include `buf180`.
2. **Stage 4 (last-use).** [`compute_last_usage`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/scheduler.py#L6811) walks in reverse. The CuteDSL node, near the end of the schedule, doesn't claim `buf180` (because Stage 2 didn't list it), so `buf180` is *not* added to `future_used_buffers`. Earlier in the iteration, the four Triton kernels — the *real* declared consumers of `buf180` — see "no later node uses this name" and one of them claims `buf180` as its `last_usage`.
3. **Stage 5 (wrapper).** The wrapper's [`codegen_free`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/codegen/wrapper.py#L3691) emits `del buf180` immediately after that Triton kernel (line 5587 of the failing output), and 130 lines later — when the CuteDSL kernel is reached — the rendered `cutedsl_fused.run(...)` has `buf180` in its argument list and Python raises `UnboundLocalError` on the first reference.

Same chain explains `arg104_1` and `arg5_1`: graph inputs the CuteDSL template doesn't list as inputs but inlines via captures. Their `del` is via `FreeLine` (input-buffer branch of `codegen_free`), but the cause is identical.

There's an alias subtlety worth flagging. `buf180` is the backing storage for ~28 `reinterpret_tensor` aliases (`buf152`–`buf179`) that the Triton kernels actually read. [`used_or_aliased_buffer_names`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/scheduler.py#L696) walks one alias hop, so the Triton kernels' alias reads do bubble up to `buf180` — that's why their `read_writes` is correct. The CuteDSL kernel reads `buf180` directly, no alias, so the alias walk doesn't help there; the underreporting happens in the bare read set, not in alias resolution.

### Self-contained reproduction

It took some work to coax a tiny graph into the same scheduling order — Inductor's topological sort tends to push CuteDSL kernels to the front, which hides the bug. The smallest version I have that fails is ~30 lines, with no external model dependencies:

```python
import torch
from torch.nn.attention.flex_attention import create_block_mask, flex_attention

device = "cuda"
dtype = torch.bfloat16
B, H, Q_LEN, KV_LEN, D = 2, 4, 256, 512, 128


def model(q, k, v, rule_masks_per_query, prefix_mask):
    masks = rule_masks_per_query & prefix_mask.unsqueeze(0)   # bitwise_and -> internal buf
    def mask_mod(b, h, q_idx, kv_idx):
        return masks[q_idx, b, kv_idx]                        # captures `masks`
    block_mask = create_block_mask(
        mask_mod, B=B, H=None, Q_LEN=Q_LEN, KV_LEN=KV_LEN, device=device,
    )
    return flex_attention(
        q, k, v, block_mask=block_mask,
        kernel_options={"BACKEND": "FLASH"},
    )


compiled = torch.compile(model, dynamic=False)
q = torch.randn(B, H, Q_LEN, D, device=device, dtype=dtype)
k = torch.randn(B, H, KV_LEN, D, device=device, dtype=dtype)
v = torch.randn(B, H, KV_LEN, D, device=device, dtype=dtype)
rule_masks = torch.randint(0, 2, (Q_LEN, B, KV_LEN), dtype=torch.bool, device=device)
prefix     = torch.randint(0, 2, (B, KV_LEN), dtype=torch.bool, device=device)

out = compiled(q, k, v, rule_masks, prefix)   # UnboundLocalError on `arg1_1`
```

The two essentials: (1) the captured tensor (`masks`) is *built* by an upstream Triton-eligible op (`bitwise_and`), so Inductor schedules a Triton kernel to produce it before flex_attention; (2) `mask_mod` reads that tensor by indexing it (`masks[q_idx, b, kv_idx]`), pulling the underlying storage and the inputs that fed the bitwise_and into the cutedsl kernel signature as captures.

Running the script under `TORCH_COMPILE_DEBUG=1` produces a wrapper whose `call(args)` body looks like this (reformatted, comments mine):

```python

    arg0_1, arg1_1, arg2_1, arg3_1, arg4_1 = args     # rule_masks, prefix, q, k, v
    args.clear()
    with torch.cuda._DeviceGuard(0):
        ...
        # Triton kernel produces buf0 from arg0_1 + arg1_1 (bitwise_and + reductions)
        triton_red_fused_arange_bitwise_and_index_permute_sum_unsqueeze_view_0.run(
            arg1_1, arg0_1, buf0, ...)
        del arg0_1                                    # <-- wrapper thinks last use was above
        del arg1_1                                    # <-- same
        ...
        # ~22 lines later, the cutedsl kernel:
        cutedsl_fused__to_copy_bitwise_and_clone_eq_flex_attention_..._ea758616.run(
            arg2_1, arg3_1, arg4_1,                   # q, k, v (named template inputs)

```

Python raises `UnboundLocalError: cannot access local variable 'arg1_1'` on the first deleted local it reads. With this script as a fixture you can step through `compute_last_usage` (Stage 4 above) and watch `arg0_1` / `arg1_1` *not* end up in `future_used_buffers` after the cutedsl node is processed — direct evidence of the Stage-2 underreport.

### The fix

Two layers, same bug. The conservative fix lives in [`torch/_inductor/ir.py`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/ir.py): override [`extract_read_writes`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/ir.py#L5378) (or extend [`_read_deps_from_inputs`](https://github.com/pytorch/pytorch/blob/e2f06e4e10823421ca984267f673f45e24018856/torch/_inductor/ir.py#L5360)) on `CuteDSLTemplateBuffer` to additionally walk the score-mod / mask-mod subgraphs and synthesize a `MemoryDep` for every captured outer-graph buffer. Sketch:

```python
class CuteDSLTemplateBuffer(TemplateBuffer):
    def extract_read_writes(self, normalize=False):
        rw = super().extract_read_writes(normalize)
        # subgraphs is the list of score_mod / mask_mod Subgraphs
        for sg in self.subgraphs or []:
            for node in sg.graph_module.graph.nodes:
                if node.op == "placeholder":
                    continue
                # Captured outer-graph buffers appear as get_attr or as
                # get_attr-resolved tensors referenced in node args.
                for cap_name in extract_captured_buffer_names(node):
                    rw.reads.add(
                        dependencies.MemoryDep(
                            cap_name, sympy.Integer(0), var_names=(), size=()
                        )
                    )
        return rw
```

The structural fix is to plumb captures through `self.inputs` directly: when building `CuteDSLTemplateBuffer`, append captured outer-graph tensors to `inputs` so they're first-class template inputs and `_read_deps_from_inputs` already sees them. That's more invasive but removes the divergence permanently.

Either fix would make the SchedulerNode's `read_writes` reflect the kernel's true reads, push `buf180`/`arg104_1`/`arg5_1` into `future_used_buffers` in Stage 4, prevent the upstream Triton kernels from claiming them in their `last_usage`, and remove the spurious `del`s from the Stage 5 output.

### Workarounds in the meantime

If you're hitting `UnboundLocalError` in inductor wrapper code with `BACKEND='FLASH'` and any captured tensor in your `mask_mod` / `score_mod`:

* Drop `kernel_options={'BACKEND': 'FLASH'}` to use the default Triton flex backend (no CuteDSL template, no bug).
* Wrap the `flex_attention` call in a `torch.compiler.disable`'d helper so its scheduling is in a separate compiled region from the upstream Triton consumers.
* Or shape the model so captures coincide with named inputs of nearby ops the scheduler already tracks correctly.

## Closing

The Inductor pipeline is a chain of layers, each consuming the previous layer's view of "what does this kernel touch?":

* **Stage 1 (lowering)** registers buffers and assigns names.
* **Stage 2 (scheduler)** computes a `ReadWrites` per node — symbolically for `ComputedBuffer`, by declared `inputs` for `TemplateBuffer`. *The* fork.
* **Stage 3 (fusion)** unions / rewrites those sets across fused groups.
* **Stage 4 (memory planning)** turns the union into per-node `last_usage` and reuse decisions.
* **Stage 5 (wrapper)** renders allocations, kernel calls, and `del`s.
* **Stage 6 (kernel codegen)** uses the same dep set to build kernel signatures.

When all layers agree, you get a clean wrapper with sensible reuse and zero memory leaks. When Stage 2 lies, the lie cascades unchanged through every later layer until something — a `UnboundLocalError`, a silent stale-data bug, a peak-memory regression — surfaces it. The DV365 case study is just one shape of "Stage 2 lied"; once you internalize the framework you can look at any inductor-generated wrapper and reason backward from a suspicious `del` to which scheduler node forgot to declare a read.

If you've made it this far and want to go deeper, the parts I cut for length: how `MultiOutputLayout` templates compose with the `_read_deps_from_inputs` shortcut; the role of `mutation_real_name` and how in-place ops work without breaking liveness; how `MemoryPlanning`'s peak-memory estimates feed back into `topological_sort_lpmf` to reorder the schedule; and the C++ wrapper path (`AOTInductor`) which has its own quirks on top of all of the above.
