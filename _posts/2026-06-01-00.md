## The question

A GRPO run starts with one command:

```bash
python -m verl.trainer.main_ppo \
    actor_rollout_ref.model.path=Qwen/Qwen3-1.7B \
    trainer.nnodes=1 trainer.n_gpus_per_node=8 \
    reward.reward_model.enable=True \
    reward.reward_model.enable_resource_pool=True \
    reward.reward_model.nnodes=1 reward.reward_model.n_gpus_per_node=8
```

A few seconds later, sixteen GPUs across two nodes are running: eight hosting a Qwen3-1.7B actor (FSDP-sharded for training, vLLM-resident for rollout), eight hosting a Qwen3-32B reward judge behind an HTTP endpoint. **Nobody wrote `ray.remote()` for those workers in the launch script.** How does verl get from a flat Hydra config to a live, GPU-pinned, distributed worker topology?

The whole path is three files:

| File | Role |
|---|---|
| [`verl/trainer/main_ppo.py`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/main_ppo.py) | Entry point. Decides *which roles exist* and *which pool each lands in*. |
| [`verl/trainer/ppo/ray_trainer.py`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/ppo/ray_trainer.py) | `RayPPOTrainer.init_workers()` — turns the plan into placement groups + worker groups. |
| [`verl/single_controller/ray/base.py`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py) | The Ray plumbing — placement groups, the `.remote()` launch, and the colocation trick. |

Here is the call chain we are about to walk, top to bottom:

```

└─ run_ppo()                                   main_ppo.py:52
   ├─ ray.init(runtime_env=...)                main_ppo.py:80      # cluster up

      └─ TaskRunner.run()                       main_ppo.py:223
         ├─ add_actor_rollout_worker()          main_ppo.py:126    # role -> class, role -> pool
         ├─ add_critic_worker()                 main_ppo.py:148
         ├─ add_reward_model_resource_pool()    main_ppo.py:193
         ├─ init_resource_pool_mgr()            main_ppo.py:158    # build pool spec
         └─ RayPPOTrainer.init_workers()        ray_trainer.py:775
            ├─ create_resource_pool()           base.py:192        # reserve GPU bundles
            ├─ create_colocated_worker_cls()    base.py:986        # fuse roles into 1 actor class
            ├─ RayWorkerGroup(...)              base.py:424        # <-- launches the actors

```

The rest of this post is that tree, one layer at a time.

## 1. `main()` → `run_ppo()`: bring up Ray, then get off the head node

The Hydra entry point is tiny — [`main_ppo.py:39-48`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/main_ppo.py#L39-L48). It resolves the device, migrates any legacy reward config, and hands off to `run_ppo`.

`run_ppo` does two things ([`main_ppo.py:52-102`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/main_ppo.py#L52-L102)). First, it brings up Ray if it isn't already, injecting a curated runtime environment (tokenizer parallelism, NCCL/vLLM log levels, LoRA flags):

```python
# main_ppo.py:62-80
if not ray.is_initialized():
    default_runtime_env = get_ppo_ray_runtime_env()
    ...
    ray.init(**OmegaConf.to_container(ray_init_kwargs))
```

Second — and this is the part worth pausing on — it runs the *driver logic itself* as a Ray actor ([`main_ppo.py:82-102`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/main_ppo.py#L82-L102)):

```python
# main_ppo.py:83
task_runner_class = ray.remote(num_cpus=1)(TaskRunner)  # please make sure main_task is not scheduled on head
...
# main_ppo.py:101-102
runner = task_runner_class.remote()
ray.get(runner.run.remote(config))
```

`TaskRunner` is the thing that orchestrates the whole run, and verl deliberately makes it a `num_cpus=1` remote actor rather than running it inline on the launching process. The inline comment — *"please make sure main_task is not scheduled on head"* — is the why: the orchestrator does heavy CPU work (tokenizer, dataset construction, the `fit()` loop driving thousands of RPCs) and you don't want it contending with the Ray GCS/dashboard on the head node. Everything from here runs *inside* that actor.

## 2. `TaskRunner.run()`: register roles, don't start anything yet

`TaskRunner` ([`main_ppo.py:111`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/main_ppo.py#L111-L124)) holds exactly two dictionaries:

```python
# main_ppo.py:122-124
def __init__(self):
    self.role_worker_mapping = {}   # Role -> ray.remote(WorkerClass)
    self.mapping = {}               # Role -> resource-pool name
```

`role_worker_mapping` says *what code* runs for each role; `mapping` says *which pool of GPUs* it lands on. The first half of `run()` ([`main_ppo.py:244-252`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/main_ppo.py#L244-L252)) just fills these in. Nothing is launched yet — this is pure bookkeeping.

### 2a. The actor (and rollout, and ref) are one class

[`add_actor_rollout_worker()`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/main_ppo.py#L126-L146) makes the central decision of a GRPO run:

```python
# main_ppo.py:140-145
if need_reference_policy(config) and not ref_in_actor:
    role = Role.ActorRolloutRef
else:
    role = Role.ActorRollout
self.role_worker_mapping[role] = ray.remote(actor_rollout_cls)   # ActorRolloutRefWorker
self.mapping[role] = "global_pool"
```

For a standard GRPO run (KL loss on, no LoRA), the role is `Role.ActorRolloutRef` and the class is a single `ActorRolloutRefWorker`. **Three logical roles — the policy being trained, the vLLM rollout engine, and the frozen reference policy for KL — are fused into one worker class that lands on `global_pool`.** This fusion is the code-level meaning of `hybrid_engine`: the same process owns the FSDP shards *and* the vLLM engine *and* the reference log-prob path, so the same weights can serve all three without copying across processes. (Note `ray.remote(actor_rollout_cls)` wraps the class but does *not* instantiate it — `role_worker_mapping` stores an actor *class*, not an actor.)

### 2b. Critic registered but dormant

[`add_critic_worker()`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/main_ppo.py#L148-L156) registers a `TrainingWorker` for `Role.Critic` on `global_pool`. GRPO has no critic — that's the whole point of the algorithm — so `use_critic` is `False` downstream and this role never actually spawns. It's registered unconditionally because `main_ppo.py` is shared by PPO, GRPO, and friends; the *trainer* decides per-run which registered roles to instantiate.

### 2c. The reward model is a pool, not a worker

This is the subtlest line in the file. [`add_reward_model_resource_pool()`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/main_ppo.py#L193-L203):

```python
# main_ppo.py:197-203
if config.reward.reward_model.enable:
    # we do not use reward model workers, so we only register reward model in resource pool
    # without continue to register reward model worker in role mapping
    if config.reward.reward_model.enable_resource_pool:
        self.mapping[Role.RewardModel] = "reward_pool"
    else:
        self.mapping[Role.RewardModel] = "global_pool"
```

Notice what's *missing*: there is no `role_worker_mapping[Role.RewardModel]`. The reward model gets an entry in `mapping` (so the resource pool is sized for it) but **no worker class**. That's because verl doesn't drive the reward model as a `WorkerGroup` of RPC methods — it stands it up as a fleet of vLLM async servers behind an OpenAI-compatible `/v1/chat/completions` endpoint, managed by the `RewardLoopManager`, and talks to it over HTTP. (The *why* — async overlap, crash isolation, externalization — is covered in the [architecture section]({{ site.baseurl }}/posts/2026-05-19-00#reward-http-not-direct-call) of the GRPO post.) The reward pool exists here only to *reserve the GPUs*; the servers are started later.

### 2d. Sizing the pools

[`init_resource_pool_mgr()`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/main_ppo.py#L158-L191) turns the config into a spec — a dict from pool name to a list of per-node GPU counts:

```python
# main_ppo.py:161-173
global_pool_id = "global_pool"
resource_pool_spec = {
    global_pool_id: [config.trainer.n_gpus_per_node] * config.trainer.nnodes,
}
if config.reward.reward_model.enable_resource_pool:
    ...
    reward_pool = [config.reward.reward_model.n_gpus_per_node] * config.reward.reward_model.nnodes
    resource_pool_spec["reward_pool"] = reward_pool
```

For the launch command at the top, this is `{"global_pool": [8], "reward_pool": [8]}` — two pools, eight GPUs each, one node each. The list-of-ints encoding is *GPUs per node*; `[8, 8]` would mean two nodes of eight. It then constructs the [`ResourcePoolManager`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L181-L217) with that spec and the role→pool `mapping`. Still nothing launched — the manager holds a *plan*.

The back half of `run()` ([`main_ppo.py:263-315`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/main_ppo.py#L263-L315)) builds the tokenizer and datasets, constructs the `RayPPOTrainer` with both dictionaries and the `RayWorkerGroup` class, and then — finally — calls the two lines that do the real work:

```python
# main_ppo.py:312-315
trainer.init_workers()   # <-- workers come alive here
trainer.fit()            # <-- training loop
```

## 3. `init_workers()`: plan → placement groups → worker groups

[`RayPPOTrainer.init_workers()`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/ppo/ray_trainer.py#L775-L895) is where the topology materializes. Three phases.

**Phase A — reserve the GPUs** ([`ray_trainer.py:782`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/ppo/ray_trainer.py#L782)):

```python
self.resource_pool_manager.create_resource_pool()
```

We'll open this in §4 — it's where Ray placement groups get created.

**Phase B — bucket worker classes by pool** ([`ray_trainer.py:784-836`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/ppo/ray_trainer.py#L784-L836)). For each role that should actually run, verl wraps its class in a `RayClassWithInitArgs` (class + the constructor args it'll eventually be called with) and files it under its resource pool:

```python
# ray_trainer.py:787-796
actor_role = Role.ActorRolloutRef if Role.ActorRolloutRef in self.role_worker_mapping else Role.ActorRollout
if self.hybrid_engine:
    actor_rollout_resource_pool = self.resource_pool_manager.get_resource_pool(actor_role)
    actor_rollout_cls = RayClassWithInitArgs(
        cls=self.role_worker_mapping[actor_role],
        config=self.config.actor_rollout_ref,
        role=str(actor_role),
    )
    self.resource_pool_to_cls[actor_rollout_resource_pool][str(actor_role)] = actor_rollout_cls
```

The result is `resource_pool_to_cls`: a map from each `RayResourcePool` to `{role_name: RayClassWithInitArgs}`. For GRPO this is `{ global_pool: {"Role.ActorRolloutRef": <cia>} }` — the critic and reward-model roles drop out (critic unused, reward model has no worker class).

**Phase C — the spawn loop** ([`ray_trainer.py:861-871`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/ppo/ray_trainer.py#L861-L871)). This is the heart of worker startup:

```python
# ray_trainer.py:861-871
for resource_pool, class_dict in self.resource_pool_to_cls.items():
    if not class_dict:
        continue
    worker_dict_cls = create_colocated_worker_cls(class_dict=class_dict)   # fuse roles -> 1 actor class
    wg_dict = self.ray_worker_group_cls(                                    # <-- LAUNCHES the actors
        resource_pool=resource_pool,
        ray_cls_with_init=worker_dict_cls,
        **wg_kwargs,
    )
    spawn_wg = wg_dict.spawn(prefix_set=class_dict.keys())                  # per-role views
    all_wg.update(spawn_wg)
```

Three calls, in order:

1. `create_colocated_worker_cls` fuses every role assigned to this pool into a single `WorkerDict` actor class (§6).
2. Constructing `RayWorkerGroup(...)` is the step that **actually creates the Ray actors** on the pool's GPUs (§5). By the time this returns, the processes exist.
3. `spawn(prefix_set=...)` hands back one *view* per role — a worker-group handle whose methods route to that role's code inside the shared actor (§5, end).

Finally, **Phase D — load the models** ([`ray_trainer.py:884-895`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/trainer/ppo/ray_trainer.py#L884-L895)):

```python
# ray_trainer.py:893-895
# we should create rollout at the end so that vllm can have a better estimation of kv cache memory
self.actor_rollout_wg = all_wg[str(actor_role)]
self.actor_rollout_wg.init_model()
```

`init_model()` is a group RPC: it runs on *every* actor in the group, loading weights, building the FSDP shards, and standing up the vLLM engine. The comment captures a real ordering constraint — vLLM sizes its KV cache from *free* GPU memory, so you want the FSDP allocations to have already happened when vLLM probes. Creating the rollout last means its KV-cache estimate reflects reality. (This is also why `gpu_memory_utilization` is so load-bearing for the rollout straggler tail discussed in the [GRPO step anatomy]({{ site.baseurl }}/posts/2026-05-19-00#anatomy-of-one-grpo-step).)

## 4. Placement groups: turning a spec into reserved GPUs

[`ResourcePoolManager.create_resource_pool()`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L192-L213) builds one `RayResourcePool` per spec entry and checks the cluster can satisfy them:

```python
# base.py:205-213
resource_pool = RayResourcePool(
    process_on_nodes=process_on_nodes,     # e.g. [8]
    use_gpu=True,
    max_colocate_count=self.max_colocate_count,   # 3 for FSDP: actor_ref, rollout, reward
    name_prefix=resource_pool_name,
)
self.resource_pool_dict[resource_pool_name] = resource_pool
...
self._check_resource_available()
```

`_check_resource_available()` ([`base.py:223-239`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L223-L239)) sums GPUs across nodes and raises early if the cluster is too small — this is the friendly "Total available GPUs X is less than total desired GPUs Y" error, rather than a deadlock waiting for resources that will never come.

The actual Ray reservation happens lazily, in [`RayResourcePool.get_placement_groups()`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L130-L160):

```python
# base.py:143-157
bundle = {"CPU": self.max_colocate_count}
if self.use_gpu:
    bundle[device_name] = 1                      # one GPU per bundle
pg_scheme = [[bundle.copy() for _ in range(process_count)] for process_count in self._store]
...
pgs = [
    placement_group(bundles=bundles, strategy=strategy, name=pg_name_prefix + str(idx), lifetime=lifetime)
    for idx, bundles in enumerate(pg_scheme)
]
ray.get([pg.ready() for pg in pgs])              # block until the scheduler reserves them
```

A few things to read out of this:

- **One bundle = one rank = one GPU.** A pool of `[8]` becomes one placement group of 8 bundles, each `{"GPU": 1, "CPU": 3}`.
- **`strategy="STRICT_PACK"`** (passed by the worker group, §5) forces all bundles of a group onto a *single node*. That's what keeps an 8-GPU FSDP world on one box with NVLink between shards, rather than scattered across the cluster.
- **`ray.get([pg.ready()])` blocks.** Until Ray's scheduler has physically reserved the bundles, this call doesn't return. When you see a GRPO job "hang at startup," it is often here — waiting on `volcano`/the autoscaler to free the bundles (the `infeasible resource request` warning).

After this, the GPUs are reserved but *empty* — no processes on them yet.

## 5. `RayWorkerGroup`: the actual `.remote()` launch

Back in the spawn loop, `RayWorkerGroup(resource_pool=..., ray_cls_with_init=worker_dict_cls)` is constructed. Its [`__init__`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L424-L489) dispatches straight into worker creation:

```python
# base.py:482-489
else:
    self._init_with_resource_pool(
        resource_pool=resource_pool,
        ray_cls_with_init=ray_cls_with_init,
        bin_pack=bin_pack,
        detached=detached,
        worker_env=self.customized_worker_env,
    )
```

[`_init_with_resource_pool()`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L536-L579) grabs the placement groups (with `STRICT_PACK` if `bin_pack`), then loops over every bundle and creates one worker per rank:

```python
# base.py:556-579
pgs = resource_pool.get_placement_groups(strategy=strategy, device_name=self.device_name)
world_size = resource_pool.world_size
...
rank = -1
local_world_size = resource_pool.store[0]
for pg_idx, pg in enumerate(sort_placement_group_by_node_ip(pgs)):
    if pg_idx == 0:
        self._get_master_addr_port(pg, bundle_index=0, ...)   # rank-0 picks MASTER_ADDR/PORT
    for local_rank in range(local_world_size):
        rank += 1
        self._create_worker(rank=rank, pg_idx=pg_idx, pg=pg, local_rank=local_rank, ...)
```

Two details worth flagging: ranks are assigned **after sorting placement groups by node IP**, so global rank order is stable and contiguous within a node; and **rank 0 elects the `MASTER_ADDR`/`MASTER_PORT`** that every other worker will use to rendezvous for `torch.distributed`.

[`_create_worker()`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L621-L680) is where a single Ray actor is born. It does three things.

**(i) Compose the distributed-init environment** ([`base.py:630-638`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L630-L638)):

```python
# base.py:630-638
env_vars = {
    "WORLD_SIZE": str(world_size),
    "RANK": str(rank),
    "WG_PREFIX": self.name_prefix,
    "WG_BACKEND": "ray",
    "RAY_LOCAL_WORLD_SIZE": str(local_world_size),
    "MASTER_ADDR": self._master_addr,
    "MASTER_PORT": self._master_port,
}
```

This is the bridge from Ray to PyTorch. Each actor reads these on startup to call `torch.distributed.init_process_group()` — verl doesn't use `torchrun`; the worker group *is* the launcher, and it injects the same env vars `torchrun` would.

**(ii) Compute the fractional GPU share** ([`base.py:627`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L627)):

```python
num_gpus = 1 / resource_pool.max_colocate_count
```

A bundle has one whole GPU, but up to `max_colocate_count` worker groups may share it (actor_ref / rollout / reward in the FSDP recipe). Requesting `1/N` of a GPU per actor lets Ray's scheduler co-place them on the same physical device — the scheduling-level half of colocation (the object-level half is §6).

**(iii) Launch the actor on its bundle** ([`base.py:673-680`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L673-L680)):

```python
# base.py:673-679
worker = ray_cls_with_init(
    placement_group=pg,
    placement_group_bundle_idx=local_rank,
    use_gpu=self.use_gpu,
    num_gpus=num_gpus,
    device_name=self.device_name,
)
self._workers.append(worker)
```

`ray_cls_with_init(...)` invokes [`RayClassWithInitArgs.__call__`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L366-L413), which finally issues the literal Ray actor creation, pinned to a specific bundle:

```python
# base.py:394-413 (essential lines)
options = {
    "scheduling_strategy": PlacementGroupSchedulingStrategy(
        placement_group=placement_group, placement_group_bundle_index=placement_group_bundle_idx
    )
}
options.update(self._options)
if use_gpu and device_name == "cuda":
    options["num_gpus"] = num_gpus
...
return self.cls.options(**options).remote(*self.args, **self.kwargs)
```

The `.remote()` at [`base.py:413`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L413) **is the moment a worker process is created** — on GPU bundle `local_rank` of placement group `pg_idx`, with its rank/world-size env already set. Run the loop eight times and you have an 8-way FSDP world, pinned one-actor-per-GPU on a single node.

When the constructor returns, `RayWorkerGroup` also binds the worker's methods onto itself ([`base.py:491-492`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L491-L492)), so the driver can call `actor_rollout_wg.generate_sequences(batch)` and have it fan out to all eight actors as one RPC. Finally, [`spawn()`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L716-L749) produces one handle per role by re-binding the prefixed methods (`ActorRolloutRef_generate_sequences` → `generate_sequences`) onto a fresh group view — so each role gets a clean API surface over the *same* underlying actors.

## 6. `create_colocated_worker_cls`: three roles, one process

The last piece is the object-level colocation. [`create_colocated_worker_cls()`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L986-L1027) dynamically builds a `WorkerDict` class that *contains* an instance of every role assigned to the pool:

```python
# base.py:1006-1018
class WorkerDict(worker_cls):
    def __init__(self):
        super().__init__()
        self.worker_dict = {}
        for key, user_defined_cls in cls_dict.items():
            user_defined_cls = _unwrap_ray_remote(user_defined_cls)
            with temp_env_var("DISABLE_WORKER_INIT", "1"):
                self.worker_dict[key] = user_defined_cls(
                    *init_args_dict[key].get("args", ()), **init_args_dict[key].get("kwargs", {})
                )
```

Then it monkey-patches each inner class's methods onto `WorkerDict` under a role prefix ([`base.py:1021-1027`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L1021-L1027)) and wraps the whole thing as `ray.remote(WorkerDict)`. That fused class is what §5 actually launches.

Two things make this work cleanly:

- **`DISABLE_WORKER_INIT=1`** during construction means the inner workers register themselves but defer all the heavy lifting (CUDA init, weight load). The real initialization is deferred to the explicit `init_model()` call in Phase D — which is why placement, colocation, and KV-cache estimation can all be ordered correctly.
- **Method prefixing** is what `spawn()` later unwinds: inside the fused actor, the actor's `generate_sequences` and the ref policy's `compute_log_prob` coexist under distinct prefixes, and each spawned worker-group view exposes just one role's methods.

For a GRPO run with `hybrid_engine`, the actor/rollout/ref are *already* one `ActorRolloutRefWorker`, so the `class_dict` for `global_pool` has a single entry and the colocation machinery is nearly a pass-through. Where it earns its keep is the FSDP recipe with `max_colocate_count=3`, where a separate actor, rollout, and reward worker genuinely share each GPU — three Python objects, one process, one CUDA context, fractional GPU each.

## Putting it together

The whole startup is a deferral pipeline — each layer records a plan and pushes the actual resource commitment one step later, so that by the time a process is created, its rank, its GPU, and its peers are all already decided:

| Layer | Artifact produced | GPU committed? |
|---|---|---|
| `add_*_worker` / `mapping` | role → class, role → pool name | no |
| `init_resource_pool_mgr` | `resource_pool_spec` = `{pool: [gpus/node]}` | no |
| `create_resource_pool` → `get_placement_groups` | Ray placement groups (bundles reserved) | **reserved**, empty |
| `create_colocated_worker_cls` | a fused `WorkerDict` actor class | no |
| `RayWorkerGroup(...)` → `_create_worker` → `.remote()` | live Ray actors, one per GPU bundle, rank env set | **processes exist** |
| `actor_rollout_wg.init_model()` | weights loaded, FSDP sharded, vLLM engine up | **GPUs in use** |

And the topology this produces for the launch command at the top:

- **`global_pool`** — 8 actors on node A, one per GPU, an 8-way FSDP world. Each is an `ActorRolloutRefWorker`: FSDP shards for the gradient step, a vLLM engine for rollout, the reference path for KL — same weights, three jobs, run in sequence (which is exactly why GPU utilization oscillates within a step, as the [step anatomy]({{ site.baseurl }}/posts/2026-05-19-00#anatomy-of-one-grpo-step) shows).
- **`reward_pool`** — 8 GPUs on node B reserved for the Qwen3-32B judge, stood up as vLLM HTTP servers by the `RewardLoopManager` (no `WorkerGroup`, no rank env — it's a service, not a distributed-training role).

That's the bridge from a one-line `python -m verl.trainer.main_ppo` to sixteen GPUs doing coordinated work. The math of GRPO is a few lines; the machinery that schedules it onto a cluster is this — and almost all of it is *deferral*, so that the single `.remote()` at [`base.py:413`](https://github.com/verl-project/verl/blob/9f73954a87e247de4c31bc2f5222969e395b2904/verl/single_controller/ray/base.py#L413) can place a worker with everything it needs already decided.
